Project TRACE — How it works, end to end

This document is the canonical reference for Project TRACE (Tracking, Routing, and Analytics Computing Engine) for the Pamantasan ng Lungsod ng Pasig (PLP) Registrar's Office. It outlines the core digital workflow (Window 1 ➔ Secretary ➔ Window 1), the AI-powered text extraction (EasyOCR), and the n8n dynamic routing logic. If the product and this document disagree, treat this document as intent and flag the gap.

Design principles

Automation over data entry

Clerks should not type what a computer can read. Python EasyOCR automatically extracts Student Numbers and Document Types from uploaded forms to eliminate manual encoding errors.

Dynamic paths, not hardcoded logic

Institutional rules change. n8n workflow webhooks handle all conditional routing, meaning the routing logic can be updated without rebuilding the core app's source code.

Streamlined Workflows

The system maps directly to reality. It focuses strictly on optimizing the core loop: Window 1 (Intake) to the College Secretary (Approval) and back to Window 1 (Release).

Unified Digital Interface

Built as a React web application, providing a clean, unified portal for students tracking documents, and a focused desktop dashboard for clerks processing them.

1. The Tech Stack & Justifications

Project TRACE utilizes a decoupled microservice architecture. Here is exactly what each technology does and why it was chosen.

  • React.js (Vite) + Tailwind CSS (Frontend): React is used to build the dynamic, single-page UI for both the student portal and the clerk dashboard. It allows for instant screen updates without refreshing the page. Tailwind is used for rapid, modern styling (Glassmorphism).
  • Node.js + Express.js (Backend API): Acts as the "API Gateway" or middleman. It securely receives HTTP requests from React and talks to the database.
  • Multer (Middleware): A Node.js middleware exclusively used for handling `multipart/form-data`. It safely catches the image files uploaded by clerks/students and stores them on the server so the AI can read them.
  • Python, Flask, & EasyOCR (AI Microservice): This is the brain of the operation. Python and Flask create a separate mini-server just for AI. EasyOCR (built on PyTorch) is a modern Deep Learning tool that is much more accurate than legacy Tesseract at reading messy, real-world academic documents.
  • MySQL (Database): The relational database that acts as the single source of truth. It stores student data, tracks which desk currently holds a document, and logs timestamps.
  • n8n (Orchestration Engine): A node-based automation tool. Instead of writing complex "If/Else" routing rules inside Node.js, we send a webhook to n8n. n8n visually evaluates the document type and automatically routes it to the Secretary's queue.

3. Student Portal & Tracking

The citizen-facing side of the app. Students do not need to download an app; it operates entirely via a mobile-responsive React web view.

  • Submission & Payment: Students authenticate via Student ID, select the credential needed, upload supporting documents, and securely pay the processing fee via automated gateway (QRPh / GCash). The document only enters the processing queue after payment is verified.
  • Tracking Link: Upon submission and payment confirmation, the system generates a unique hash and sends a tracking URL. The student can view a linear timeline of exactly whose desk their document is on.

4. Clerk Dashboard (React UI)

The primary work surface. A clean UI that eliminates clutter and focuses strictly on the active queue.

  • The Queue: A real-time data table fetching from the Node.js backend. Displays only the documents currently assigned to that specific clerk (e.g., Window 1 only sees Intake/Release, Secretary only sees Approvals).
  • Split-Screen Verification: When a clerk opens a file, they see the uploaded image on the left, and the AI-extracted text on the right.
  • Quick Actions: Green Approve and Red Return buttons trigger API calls that advance the state and instantly remove the item from the local view.

5. Intelligent Intake (Deep Learning AI)

To eliminate transcription errors, TRACE uses a Python microservice for automated data entry.

Workflow

When an image or PDF is uploaded by Window 1, Node.js securely pipes it to the Python Flask API. EasyOCR (powered by PyTorch) reads the image utilizing Convolutional Recurrent Neural Networks (CRNN). Regex pattern matching extracts the Student Number and Form Type. This is returned as a clean JSON payload and injected straight into the MySQL database, bypassing manual typing.

6. Dynamic Routing Engine (n8n)

The "traffic cop" of the system. Hardcoded paths break when the university changes its rules. By using n8n, routing is visual and modular.

  • Webhooks: When Window 1 uploads a file, Node.js fires an event to n8n.
  • Conditional Nodes: n8n evaluates the payload. It then executes a database update, automatically changing the document's queue to the College Secretary. Once the Secretary approves, n8n routes it back to Window 1 for release.

This section is the technical build-order companion for the development team. The Phased Release Plan defines the exact sequence to build the core loop.

7.1 Data Model (MySQL)

The core tables TRACE relies on to execute routing.

Table: documents
{
  id: INT (PK, Auto-Increment),
  tracking_number: VARCHAR (Unique Hash),
  student_id: VARCHAR,
  document_type: VARCHAR,
  current_status: VARCHAR,
  payment_status: VARCHAR, // e.g., 'UNPAID', 'PAID'
  assigned_desk: VARCHAR // e.g., 'NONE', 'WINDOW_1', 'SECRETARY'
}

Table: step_logs
{
  id: INT (PK, Auto-Increment),
  document_id: INT (FK),
  desk_name: VARCHAR,
  action_taken: VARCHAR, // 'APPROVED', 'REJECTED'
  timestamp: DATETIME
}

7.2 Phased Release Plan

Follow this strict ship order. Do not deploy a later phase before its prerequisites.

  1. The Foundation & Tech Stack Handshake (Phase 1)

    Get React, Node, Python, and MySQL talking to each other.

    Build

    • Frontend: Build static React Clerk Login and Document Upload Page.
    • Backend: Spin up MySQL schemas. Write Node.js Express endpoints with Multer for file uploads (`POST /upload`). Install local n8n.
    • AI: Set up Python Flask. Write a test EasyOCR script that reads a sample image and returns raw text.

  2. Automated Payments (Phase 1.5)

    Integrate payment gateway to ensure requests are paid before processing.

    Build

    • Phase 1.5: Automated Payments & Account Verification: Webhooks for GCash/QRPh. Signup verification flows for students/alumni (including ID/Diploma uploads).
    • AI Auto-Verification: Python EasyOCR scans uploaded IDs during signup to instantly auto-verify students, falling back to manual admin review if unreadable.
    • Users: id, student_id, user_type (student/alumni), id_proof_path, role (student/clerk/admin), verification_status (pending/verified/rejected)
    • Backend: Set up webhook endpoints to listen for payment confirmation. Only route the document to Window 1 when payment is marked as PAID.

  3. Intelligent Intake & Auto-Routing (Phase 2)

    Turn raw text into intelligent sorting via n8n.

    Build

    • Frontend: Build the Clerk Queue Dashboard table fetching data from Node.
    • Backend: Build n8n webhook logic (Window 1 ➔ Secretary).
    • AI: Refine EasyOCR with Regex to isolate Student Number and Document Type. Return as clean JSON.

  4. Dynamic Hand-off & The Core Loop (Phase 3)

    Prove end-to-end pipeline functionality for the core workflow.

    Build

    • Frontend: Add Approve/Reject action buttons to the React dashboards.
    • Backend: Wire React buttons to n8n webhooks to update MySQL database states and transfer files between Window 1 and the Secretary.

    Definition of Done

    • Window 1 uploads paper -> AI extracts data -> n8n assigns it to Secretary -> Secretary clicks Approve -> n8n routes back to Window 1.
