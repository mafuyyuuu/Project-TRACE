Project TRACE — How it works, end to end

This document is the canonical reference for Project TRACE (Tracking, Routing, and Analytics Computing Engine) for the Pamantasan ng Lungsod ng Pasig (PLP) Registrar's Office. It outlines the core digital workflow, the Command-Center UI architecture, manual payment verifications, and Machine Learning forecasting.

Design principles & UI Mandates

Fluid "Command-Center" Layouts

Dashboards must maximize screen real estate. We do not use constrained white-space containers. The UI relies on fluid CSS grids spanning the full viewport to present dense, engaging KPIs.

Manual Payment Gate

Payments bypass automated gateways. Students scan a static GCash QR, then upload the receipt and reference number. Finance Clerks manually verify these before routing documents.

Automation over data entry

Clerks do not type what a computer can read. Python EasyOCR extracts Student Numbers and Document Types from uploaded forms via a split-screen verification modal.

Predictive Management

The system uses Prophet & Random Forest ML to forecast volume spikes, alerting the Admin via an aesthetically pleasing dashboard with actionable cards.

1. The Tech Stack & Justifications

Project TRACE utilizes a decoupled microservice architecture tailored for high-density, real-time interfaces.

  • React.js (Vite) + Tailwind CSS: Drives the responsive, fluid-width command-center dashboards. Uses Lucide icons and soft pine-green branding constraints.
  • Node.js + Express.js (Backend API): Securely handles uploads, routing, and serves static files (like receipt uploads).
  • Python, Flask, & EasyOCR: Deep Learning AI microservice. Reads messy, physical academic documents during Window 1 intake.
  • Python, Prophet, & Scikit-Learn: The analytical brain generating 7-day volume forecasts and prescriptive process recommendations.
  • MySQL (Database): Source of truth, logging every state change timestamp for ML training.

3. Student Portal & 3-Column Layout

The citizen-facing side is heavily optimized for aesthetics and usability, moving away from boring standard forms.

  • Unified Layout: The view uses a responsive 3-column grid. The tracker spans two columns, while the submission form sits comfortably in a sticky side-column.
  • Visual Tracking: Progress is indicated by visual bars (amber, blue, pine green) and dynamic status badges, giving the student immediate visual feedback.

4. Command-Center Clerk Dashboards

The primary work surfaces for all Clerks utilize the full width of the screen to present dense, actionable data.

  • Top-Level KPIs: Dashboards feature prominent metrics at the top (e.g., "Scanned Today", "Pending Verification") to provide instant situational awareness.
  • Interactive Queues: Clean, borderless tables with hover-states that reveal quick-action buttons (Approve/Reject/Release) without cluttering the screen.

5. The Manual Payment Workflow

To comply with university restrictions on third-party payment gateways, TRACE utilizes a secure manual verification pipeline.

Workflow

When a student initiates a request, the UI presents a static GCash QR code belonging to the PLP Finance office. The student pays via their app, then uploads a screenshot and inputs the Reference Number into TRACE. The Node.js backend flags the document as PENDING_PAYMENT_VERIFICATION. It sits in the Finance Clerk's dashboard until they manually confirm the transaction against their records and click "Approve."

6. Split-Screen OCR Evaluation (AI)

When the College Secretary clicks a pending document, it triggers the Evaluation Modal.

The modal locks the background and presents a split-screen view. The left side displays the original scanned image with simulated AI bounding boxes. The right side displays the extracted data (Student ID, Name, Type) alongside an AI Confidence Score. The Secretary verifies the fields and clicks "Approve & Route".

7. Predictive Analytics & Admin Tools

The Registrar Admin view is a dedicated analytics and operations command center.

  • Prophet Volume Forecast: A visual chart predicting incoming document volume over the next 7 days, highlighting peak rush days.
  • AI Insights Panel: A dark-themed card utilizing Random Forest heuristics to output prescriptive, plain-English advice (e.g., "Expect a 200% surge... Temporarily reassign staff to Window 1.").
  • Student Verification Desk: A list of pending student account registrations where EasyOCR auto-verification failed. Admins inspect the uploaded ID proofs and manually click "Verify" or "Reject".

8. Frequently Asked Questions (Dev)

  • How do we add a new document type to the system?

    You must update the array of accepted document types in the React frontend (e.g., const DOCUMENT_TYPES = [...]). You must also ensure the backend schema validation accepts the new string.

  • Why aren't we using PayMongo/Xendit?

    As this is a capstone/thesis implementation, acquiring live production API keys for payment gateways requires valid business registrations and school bank accounts. The Manual QR Verification flow securely simulates this process while demonstrating complex state management and role-based approvals.

  • How is the Random Forest ML model retrained?

    The model requires fresh historical data to improve its bottleneck predictions. A scheduled cron job should run monthly, triggering the Python analytics service to fetch the latest timestamps from the step_logs MySQL table, retrain the model, and save the updated `.pkl` file.

9.1 Data Model (MySQL)

Table: documents
{
  id: INT (PK, Auto-Increment),
  tracking_number: VARCHAR (Unique Hash) NOT NULL,
  student_id: VARCHAR(50) (Nullable),
  student_name: VARCHAR(255) (Nullable),
  document_type: VARCHAR(100) (Nullable),
  current_status: VARCHAR(50) DEFAULT 'pending_payment', // values: 'pending_payment', 'pending_payment_verification', 'pending_secretary', 'ready_window_1', 'completed', 'rejected'
  payment_status: ENUM('UNPAID', 'PAID') DEFAULT 'UNPAID',
  assigned_clerk_id: INT (FK, Nullable),
  file_path: VARCHAR(500) (Nullable),
  original_filename: VARCHAR(255) (Nullable),
  receipt_image_path: VARCHAR(500) (Nullable),
  payment_reference_id: VARCHAR(255) (Nullable),
  gcash_reference_no: VARCHAR(255) (Nullable),
  ocr_raw_text: TEXT (Nullable),
  ocr_extracted_data: JSON (Nullable)
}

Table: step_logs
{
  id: INT (PK, Auto-Increment),
  document_id: INT (FK) NOT NULL,
  clerk_id: INT (FK, Nullable),
  action_taken: VARCHAR(100) NOT NULL,
  from_status: VARCHAR(50) (Nullable),
  to_status: VARCHAR(50) (Nullable),
  timestamp_started: DATETIME DEFAULT CURRENT_TIMESTAMP,
  timestamp_completed: DATETIME (Nullable),
  notes: TEXT (Nullable)
}
UI Spec Adherence

The UI must strictly adhere to the "Command-Center" philosophy. Do not wrap dashboards in max-width containers. Ensure the "TRACE" branding is prominent in the header. Use the exact specified Tailwind color palette (Pine & Dusk).
